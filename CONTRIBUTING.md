# Contributing to CoreForge

## Standards
- Keep architecture explicit and readable
- No hidden framework magic
- Preserve module boundaries
- Favor one obvious way for each concern

## Contribution flow
1. Open issue or discussion
2. Submit focused PR with docs updates
3. Ensure checks pass
4. Include migration notes when schema changes

## Pull request checklist
- [ ] Code follows module conventions
- [ ] Tests added/updated where meaningful
- [ ] Docs updated
- [ ] No breaking changes without notes

## Commit style
Conventional commits preferred (feat, fix, docs, refactor, test, chore).
